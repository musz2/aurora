## 2. Bugs found & fixed this session

### LOOP 1 — Repository hygiene
- **Committed build artifacts.** `apps/desktop/dist-electron/` (compiled
  Electron `main.js`/`preload.js`, regenerated by the build script) was tracked
  in git. Untracked it and added `dist/`, `dist-electron/`, `release/` to
  `.gitignore` and `.dockerignore`.
- **Invalid workspace config.** `pnpm-workspace.yaml` contained an `allowBuilds`
  block with literal placeholder text (`set this to true or false`). Removed it;
  build approval is already handled by `pnpm.onlyBuiltDependencies` in the root
  `package.json`.
- **Stray directory.** Removed empty untracked `apps/uploads/` (the real upload
  dir is `apps/server/uploads/`, already gitignored).
- No secrets are committed. `.env` is present locally but git-ignored and
  untracked; only `.env.example` (safe placeholders) is tracked.

### LOOP 2 — Build / test
- **Hanging test suite (CI blocker).** `pnpm --filter @aurora/server test`
  passed all tests but never exited, because the `ioredis` client is
  instantiated at module import time and holds an open reconnect handle. Added
  `--test-force-exit` to the test script. Tests already passed; this only makes
  the process terminate. Confirmed stable across repeated runs (105/100/5).

### LOOP 3 — Backend hardening (security)
- **Ungated owner billing override (production risk).** `ownerBillingOverrideActive()`
  previously returned `true` whenever an email was present in the legacy
  `DEVELOPER_BYPASS_EMAILS` var — with **no** gate. Because that same var is also
  honored as the owner-allowlist alias, merely marking an admin account in
  production silently unlocked all paid features. Fixed so the override requires
  the explicit `ENABLE_OWNER_BILLING_OVERRIDE="true"` gate as the **sole**
  trigger. The allowlist now only marks operator accounts; it never unlocks paid
  features on its own. Tests updated to assert the gated, production-safe contract.
- **Production CORS trusted localhost.** `getAllowedOrigins()` always injected
  `http://localhost:5173/4173`. Now localhost is trusted only outside
  production, so a prod deploy accepts exactly its configured origins
  (`FRONTEND_URL` / `WEB_URL` / `CORS_ALLOWED_ORIGINS`).

### LOOP 10 — Billing / plan-limit enforcement
- **Unenforced usage limits.** `canStartRecording()` (monthly transcription
  minutes) and `checkConcurrentAllowed()` / `getActiveSessionCount()`
  (seat-based concurrent live sessions) existed and were unit-tested but were
  **never called** — a live meeting could always start regardless of the plan
  cap. Wired both into `POST /api/meetings/:id/start`, which now returns HTTP
  402 with an actionable message when the concurrent-session or monthly-minute
  cap is reached, before flipping the meeting to `RECORDING`.

## 3. Safety-critical properties audited and confirmed already correct

These were inspected against the project's consent-first / no-bypass rules and
found sound — no change needed:

- **Public config route** (`/api/config`) exposes only booleans (whether a
  service is configured), never keys.
- **CORS** blocks non-allowlisted origins with a clean 403 (not a silent 500).
- **Rate limiting** present: `/api` 300/min, `/api/auth` 30/min.
- **Stripe webhook** registered with `express.raw` before the JSON body parser
  for signature verification; public and Stripe-signed.
- **Env validation** fails fast only on missing `DATABASE_URL`; in production it
  warns (not crashes) on missing JWT secrets and auto-generates strong ephemeral
  secrets per boot — honest and non-crash-looping. No external provider key is
  ever required (mock/demo mode is honest).
- **Companion Mode** (LOOP 7): pairing tokens are `cmp_<nanoid(32)>`, persisted
  only as SHA-256 hashes, returned once, resolved by hash, revocable
  (`revoked` flag) and expiring (`expiresAt`), and never written to logs.
- **Share links** (LOOP 12): `s_<nanoid(12)>` (~71 bits, unguessable), with
  `shareExpiresAt` expiry; unsharing regenerates the id, invalidating old links.
- **Owner allowlist** is read server-side only and never referenced in the web
  or shared bundles.
- **Feature gating** (LOOP 10): `requireFeature`/`requirePlan` are consistently
  applied across AI chat, advanced summaries, exports, uploads, integrations,
  calendar, search history, custom vocabulary, and team workspace routes.
  Upload import caps are enforced via `canUpload`.
- **Integrations honesty** (LOOP 9): `integrations.service.ts` classifies every
  provider as connected / disconnected / needs_approval / failed / mock, runs
  live actions only when real tokens are present, refreshes OAuth tokens on
  expiry, and records `lastError` on failure. No provider is claimed live while
  mock-only.
- **Stripe webhook** (LOOP 10): resolution is pure + unit-tested; plan
  application via `upsert` is naturally idempotent on replayed events; plan is
  applied only from the webhook after payment, never optimistically at checkout.
- **Live transcription** (LOOP 4): AUDIO_READY handshake and pure final-segment
  de-duplication helpers are implemented and unit-tested; the socket keeps a
  bounded in-memory window of recent finals to skip duplicates on reconnect.

## 6. Deployment checklist (server)

- Set `NODE_ENV=production`, `DATABASE_URL`, stable `JWT_SECRET` +
  `JWT_REFRESH_SECRET`.
- Set `FRONTEND_URL` (and any `CORS_ALLOWED_ORIGINS`) — localhost is no longer
  trusted in production.
- Leave `ENABLE_OWNER_BILLING_OVERRIDE` **unset** so billing relies on real
  subscription status.
- Provide provider keys only for integrations you actually intend to run live;
  everything else stays honestly in mock/not-configured state.
